# bidding-player

A Prebid.js-powered video header-bidding player that auctions against GAM. Ships as a single hosted `player.js` engine that publishers embed with a one-line `<script>` tag.

## Layout

```
.
├── index.html              # AdOps tag generator (hosted on GitHub Pages)
├── engine/player.js        # The runtime publishers load via jsDelivr
├── prebid/prebid.js        # Prebid.js v11.11.0 build (limelightDigital adapter included)
├── demo/publisher-test.html # Example publisher page using a generated tag
├── VERSION                 # Single source of truth for the current release
└── .github/workflows/      # CI checks
```

## URLs

| Channel       | Pattern                                                                              |
| ------------- | ------------------------------------------------------------------------------------ |
| Production    | `https://cdn.jsdelivr.net/gh/shashwatsilverpush/bidding-player@vX.Y.Z/engine/player.js`    |
| Testing       | `https://cdn.jsdelivr.net/gh/shashwatsilverpush/bidding-player@<branch>/engine/player.js`  |
| Tag generator | `https://shashwatsilverpush.github.io/bidding-player/`                                     |

Prod tags are **immutable** — once `v1.0.0` is published, the bytes at that URL never change. Branch tags exist for testing and are cached by jsDelivr for ~30 min.

## Release workflow

The whole point of this repo's structure: **engine, prebid, and the demo always move together**. No more "I updated player.js on GitHub but forgot to refresh the demo page."

### Testing a change

```bash
git checkout -b feat/my-change
# edit engine/player.js (or prebid.js, or whatever)
git commit -am "feat: my change"
git push origin feat/my-change
```

Then open the tag generator (`index.html` locally, or the Pages URL), type `feat/my-change` into the **Release Version / Branch** field, click Generate, and paste the resulting `<script>` into your test page. The engine and prebid URLs both auto-point at the branch, so you're testing the full set together.

### Cutting a release

1. Make sure `main` is green and the change has been tested against a branch tag.
2. Bump the `VERSION` file (e.g. `1.0.0` → `1.0.1`).
3. Update the two default jsDelivr URLs in `index.html` to match — search for the previous version string (`@v1.0.0`) and replace with the new one. The CI check below will catch drift.
4. Commit, tag, push:

   ```bash
   git commit -am "chore: release v1.0.1"
   git tag v1.0.1
   git push origin main --tags
   ```

5. jsDelivr picks up the tag within ~1 min. Any new tag generated by AdOps now defaults to the new version.

### Rollback

A bad release does not affect publishers who already deployed the previous version's tag — their `@v1.0.0` URL still serves the old bytes. To roll back the tag generator's default:

```bash
# delete the bad tag
git tag -d v1.0.1
git push origin :refs/tags/v1.0.1
# revert VERSION and index.html defaults to v1.0.0
```

## Local development

Open `index.html` in a browser. Tab 1 is the sandbox (mock bids, verbose logs) for testing GAM key-value flow. Tab 2 generates production tags.

For the demo page to load the engine correctly while iterating locally, run a static server in the repo root:

```bash
python3 -m http.server 8000
# then open http://localhost:8000/demo/publisher-test.html
```

Add `?debug=true` to any publisher page to see the engine's STEP / WINNER / NO-DEMAND console logs.

## What's inside the engine

The runtime at `engine/player.js`:

1. Reads config from its own `<script>` tag's `data-*` attributes.
2. Lazy-loads video.js, the Google IMA SDK, and the pinned Prebid build (in parallel).
3. Runs a Prebid auction against the `limelightDigital` adapter using custom $0.10 buckets.
4. Applies a configurable floor bias (`data-bias`) and stitches `hb_pb` + `hb_bidder` into the GAM tag's `cust_params`.
5. Hands the final VAST URL to IMA for ad playback, then resumes content.

Falls back to a `$0.00` GAM call if Prebid fails, no bid comes back, or IMA is blocked.
