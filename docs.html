Publisher Integration Guide

A drop-in JavaScript tag that runs a Prebid header-bidding auction against limelightDigital, stitches the winning bid into your GAM ad call, and plays the resulting VAST creative through video.js + Google IMA.

Overview

Bidding Player lets a publisher monetize their video inventory with real-time header bidding without writing any Prebid integration code. A single <script> tag, configured with data-* attributes, handles everything:

  • Lazy-loads video.js, Google IMA SDK, and a custom Prebid build (≈ 250 KB total, parallelized).
  • Registers an instream video ad unit with the limelightDigital bidder.
  • Runs the auction against your configured timeout (default 1200 ms).
  • Stitches the full Prebid targeting key-value set (hb_pb, hb_bidder, hb_uuid, hb_size, hb_format, hb_adid, plus bidder-suffixed variants) into your GAM master ad tag's cust_params.
  • Hands the final VAST URL to IMA, which serves either the winning bid's cached creative (resolved via hb_uuid against Prebid Cache) or your house ad if no bid wins.

The engine is hosted at versioned, immutable jsDelivr URLs. A tag pinned to @v1.2.2 serves the exact bytes that were tested at release time and cannot silently change.

What's bundled

  • video.js v8 (player UI)
  • Google IMA SDK v3 (ad rendering)
  • Prebid.js v11.11.0 with the limelightDigital adapter, TCF v2 / USP consent modules

What you provide

  • A Limelight publisher ID + ad unit ID
  • Your GAM master video ad tag
  • GAM line items targeting hb_* keys
  • The content video URL

How it works

The end-to-end auction-to-impression sequence:

Browser loads the publisher page; the engine's async <script> executes.
Engine reads its own data-* attributes for runtime configuration.
Engine parallel-loads video.js, IMA SDK, and Prebid.js from CDN.
Engine measures the mount-div's rendered size; registers an instream video ad unit with limelightDigital.
Prebid → Limelight: POST OpenRTB bid request to {host}/ortbhb.
Limelight → Prebid: bid response (or no-bid).
Prebid caches the winning VAST at prebid.adnxs.com/pbc/v1/cache; receives a UUID.
Engine calls pbjs.getAdserverTargetingForAdUnitCode to obtain the full hb_* key set.
Engine applies its floor bias to hb_pb; stitches all keys into GAM's cust_params.
Engine → IMA: AdsRequest.adTagUrl = <stitched GAM URL>.
IMA → GAM: fetches VAST; matches your line item targeting hb_pb.
GAM returns a VAST wrapper whose URL contains {hb_uuid} — GAM substitutes the macro with the cached bid's UUID.
IMA → Prebid Cache: resolves the wrapper, retrieves the actual bidder VAST.
IMA plays the ad. Engine pauses content (z-index swap on ad container).
Ad completes; Engine swaps z-index back; content resumes.
Fallback path

If Prebid fails to load, the bidder times out, or no bid comes back, the engine still calls GAM with hb_pb=0.00&hb_bidder=none in cust_params. GAM's house ad / passback line item fills the impression. The publisher never loses an ad opportunity to a network failure.

Quick start

Five steps from zero to a live monetizing player.

  1. Generate your tag

    Open the Tag Generator and fill in:

    • Release version (use the latest tag, e.g. v1.2.2)
    • Your GAM Master Ad Tag URL
    • Your Limelight Publisher ID and Ad Unit ID
    • Your content video URL

    Click ⚡ Generate Production Tag and copy the resulting <script> snippet.

  2. Set up GAM line items + creative

    Create one VAST wrapper creative whose URL is your Prebid Cache + {hb_uuid} macro, and one price-bucket line item per $0.10 CPM step. See the GAM setup section for the exact configuration. (You only need to do this once per ad unit.)

  3. Embed the snippet

    Paste the generated <script> tag into your page at the position where the video player should render. The engine auto-creates the mount <div> next to the script.

  4. Smoke-test in your browser

    Open the page with ?debug=true appended to the URL. In DevTools → Console you'll see STEP 1, STEP 1.5, then either 🏆 WINNER or ⚠️ NO MARKET DEMAND, then STEP 6. In the Network tab, filter for ortbhb to see the OpenRTB bid request. See Testing & verification for the full checklist.

  5. Monitor

    Once live, monitor in GAM Reports for hb_bidder=limelightDigital impressions, eCPM, and fill rate. If you don't see Limelight impressions after 24 hours, see Troubleshooting.

Embed code reference

The canonical embed snippet. Every configurable knob is a data-* attribute on the <script> tag.

<script src="https://cdn.jsdelivr.net/gh/aryanvani-pm/bidding-player@v1.2.2/engine/player.js"
        id="adtech-player-core"
        data-pub-id="649658371"
        data-adunit-id="972556929"
        data-tag="https://pubads.g.doubleclick.net/gampad/ads?iu=/…&sz=640x480&output=vast&correlator="
        data-timeout="1200"
        data-bias="0.10"
        data-video="https://your-cdn.example.com/content.mp4"
        data-autoplay="true"
        data-muted="true"
        data-fluid="true"
        data-loop="false"
        data-preload="metadata"
        data-vpaid="insecure"
        data-div-id="comparos-video-placement"
        data-host="ads-jbi003.rtba.bidsxchange.com"
        data-cache="https://prebid.adnxs.com/pbc/v1/cache"
        data-prebid-url="https://cdn.jsdelivr.net/gh/aryanvani-pm/bidding-player@v1.2.2/prebid/prebid.js"
        async></script>

Attribute reference

Attribute Required Default Description
src Yes — Versioned jsDelivr URL of the engine. Always pin to a tag (@v1.2.2), never @main.
id Yes adtech-player-core Must equal adtech-player-core. The engine uses document.currentScript primarily, with this ID as a fallback for legacy browsers.
async Yes — Required HTML attribute. The engine boots after the DOM is ready and won't block page rendering.
data-pub-id Yes — Your Limelight publisher ID. Provided by Limelight.
data-adunit-id Yes — Your Limelight ad unit ID (numeric). Provided by Limelight.
data-tag Yes — Your full GAM Master Ad Tag URL. Must end with correlator= (no value) so the engine can refresh it per request.
data-prebid-url Yes — URL of the Prebid.js build. Must include limelightDigitalBidAdapter. The version released here at @v1.2.2/prebid/prebid.js is correct.
data-host No ads-jbi003.rtba.bidsxchange.com Limelight bidder endpoint host. Only change if Limelight directs you to.
data-video No — Content video URL (.mp4 / .m3u8). If omitted, only the pre-roll plays and the player shows a black screen after.
data-timeout No 1200 Prebid bidder timeout in milliseconds. Lower = lower latency, higher rate of timed-out bids. Industry default is 1500–2000.
data-bias No 0.10 Floor bias in USD added to the winning CPM before bucketing. Used to align bid prices with your GAM line item structure.
data-autoplay No true Whether the player attempts to autoplay on load. Browser autoplay policy requires muted=true for this to succeed without a user gesture.
data-muted No true Whether the player starts muted. Required if autoplay=true in most browsers.
data-fluid No true true = responsive 16:9 container scaling to width. false = fixed 640×480 box.
data-loop No false Loop the content video after it ends.
data-preload No metadata HTML5 video preload hint: metadata (recommended), auto, or none.
data-vpaid No insecure IMA VPAID handling: insecure (max yield, runs VPAID in same frame), enabled (cross-origin iframe), or disabled (refuses VPAID creatives).
data-div-id No comparos-video-placement ID of the mount div. If no element with this ID exists at boot, the engine creates one at the script's position. Override if you want to control placement via your own CSS.
data-cache No https://prebid.adnxs.com/pbc/v1/cache Prebid Cache server. Default is the free public AppNexus cache. Switch to your own Prebid Server cache for higher SLA and volume.
Don't change versions piecemeal

The src and data-prebid-url attributes must always point at the same version tag. The Tag Generator enforces this; if you hand-edit your snippet, keep them in sync.

GAM setup

This is the side that handles the actual ad serving. If you've done Prebid + GAM integration before, this will be familiar. If not, it requires careful one-time configuration of custom key-values, a VAST wrapper creative, and price-bucket line items.

Step 1: Create custom key-values

In GAM go to Inventory → Key-values → New key-value. Create the following keys with Value Type: Predefined (or Free-form if you'd rather not enumerate every bucket):

Key nameValue typeSuggested values
hb_pbPredefined0.00, 0.10, 0.20, …, 20.00 (one value per $0.10 step)
hb_bidderPredefinedlimelightDigital, none
hb_uuidFree-form(left free-form — UUIDs are dynamic)
hb_sizePredefined640x360, 1280x720, 1920x1080, etc.
hb_formatPredefinedvideo
hb_adidFree-form(dynamic)
hb_sourcePredefinedclient, s2s
Why bidder-suffixed keys too?

Prebid also emits hb_pb_limelightDigital, hb_uuid_limelightDigital, etc. Targeting these is optional — they let you build "first-look" or per-bidder private auction line items. You can skip them for a simple setup.

Step 2: Create the VAST wrapper creative

The creative is a single VAST URL with a macro that pulls the cache UUID at serve time:

https://prebid.adnxs.com/pbc/v1/cache?uuid=%%PATTERN:hb_uuid%%

Substitute the URL of your own Prebid Cache server if you operate one. The %%PATTERN:hb_uuid%% macro is GAM's syntax for "look up the hb_uuid key-value from the ad request and inject its value here."

To create:

  1. GAM → Delivery → Creatives → New creative
  2. Select your video ad unit's advertiser
  3. Type: Video → VAST 3.0 ad tag
  4. Paste the URL above into the VAST tag URL field
  5. Set duration to 0:00 (the wrapper resolves the real duration)
  6. Save

Step 3: Create price-bucket line items

You need one line item per $0.10 price bucket from $0.10 up to your maximum expected CPM (e.g. $20.00 = 200 line items). The structure is identical for each; only the CPM and target value change.

Per line item

FieldValue
NamePrebid HB $2.50 (or your naming convention)
Inventory sizeVideo — match your ad unit's size(s)
TypePrice Priority
CPM rateMatch the bucket exactly (e.g. $2.50)
GoalNone (unlimited) or your impression cap
Targeting → InventoryYour video ad unit
Targeting → Key-valueshb_pb is one of 2.50
CreativeThe VAST wrapper from Step 2

Use Prebid's official Line Item Generator tool to bulk-create all 200 line items via CSV upload — manual creation is impractical at this volume.

Exact-match values matter

Prebid's hb_pb values are always strings like "2.50" (two decimal places, no currency symbol). Your GAM line item targeting must match that exact string. A line item targeting "2.5" will not match "2.50" and will never serve.

Step 4: Set up house ad fallback

When no bid wins, the engine sends hb_pb=0.00&hb_bidder=none. Create one of:

  • A house line item targeting hb_pb=0.00 with a low priority and your fallback creative (an in-house promo, a sponsor card, or your own VAST passback).
  • A remnant line item targeting the same key-value pointing at AdSense / a tag-based ad network — GAM will fall through to fill the slot.
  • Leave it blank if you're OK with the slot serving nothing on no-bid (rare; most pubs want to fill 100%).

Step 5: Wrap line items in an Order

GAM requires every line item to live inside an Order. Create a single Order called e.g. "Prebid Header Bidding" and assign all 200 price-bucket line items + the fallback to it. This lets you pause / activate the entire integration with one click.

Limelight onboarding

The bidder side is owned by Limelight Digital. Steps you (or your account manager at the platform that integrated this player) follow:

  1. Apply for a Limelight account. If you don't have one, contact your platform's CSM to request publisher onboarding.
  2. Receive your credentials. Limelight provides:
    • publisherId — numeric, e.g. 649658371 → goes into data-pub-id
    • adUnitId — numeric, e.g. 972556929 → goes into data-adunit-id
    • host — bidder endpoint, e.g. ads-jbi003.rtba.bidsxchange.com → goes into data-host
  3. Submit your domain(s) for allowlist. Limelight only bids on requests from approved publisher domains. Without this step, every auction returns no-bid in production.
  4. Wait for allowlist approval. Typically 1–3 business days.
  5. Verify bidding is live using the steps in Testing & verification.
Tip

You can test the integration mechanics (script loads, auction fires, GAM call goes out, ad plays from GAM's test creative) before your domain is allowlisted. The bidder side will just return no-bid until allowlisting completes. That's why the engine has a fallback path.

Consent management

The engine ships consent-aware defaults that are appropriate for non-EU traffic but require an override for EU / California / other-state-privacy traffic. Below is what to do for each scenario.

What the engine does by default

FrameworkDefault behavior
GDPR (TCF v2)Declares gdprApplies: false via static config. The bundled tcfControl module skips enforcement. Suitable for US/India/non-EU traffic.
USP (CCPA)Auto-detects window.__uspapi with a 100 ms timeout. If no CMP responds, proceeds without USP signal.
GPP (US state privacy)Not activated. The bundled consentManagementGpp module cancels the auction outright when no GPP CMP is found, so it is intentionally left off.

EU publishers — enable real TCF v2 consent

If you serve EU traffic, you MUST place a registered TCF v2 CMP on your page (OneTrust, Sourcepoint, Quantcast, Didomi, etc.) and override the engine's static GDPR config. Place this snippet before the engine's <script> tag in your page:

<script>
window.pbjs = window.pbjs || { que: [] };
pbjs.que.push(function() {
  pbjs.setConfig({
    consentManagement: {
      gdpr: {
        cmpApi: 'iab',        // detect window.__tcfapi
        timeout: 8000,         // wait up to 8s for CMP response
        defaultGdprScope: true  // assume GDPR applies unless CMP says otherwise
      }
    }
  });
});
</script>

This must run before pbjs finalizes config in the engine's auction. Putting it just before the engine <script> tag is reliable.

California / Connecticut / Virginia (USP)

The engine auto-detects USP. Just ensure your CMP exposes window.__uspapi. To extend the timeout (the default 100 ms is fast and may miss slow CMPs):

pbjs.setConfig({
  consentManagement: {
    usp: { cmpApi: 'iab', timeout: 3000 }
  }
});

Multi-state US (GPP)

If your CMP supports GPP, opt in explicitly:

pbjs.setConfig({
  consentManagement: {
    gpp: { cmpApi: 'iab', timeout: 3000 }
  }
});
Without a CMP in regulated jurisdictions

Operating in the EU, California, or other regulated states without a registered CMP is a regulatory risk independent of this player. The engine's permissive defaults are a publisher-experience choice — they do not constitute legal compliance. Consult counsel.

Testing & verification

Before declaring the integration live, verify each of the following.

Enable debug mode

Append ?debug=true to your page URL. The engine logs the auction flow in the browser console with color-coded steps.

Console checklist

Open DevTools → Console. Filter for STEP. You should see:

  • STEP 0.5 Mount point auto-created (only if you didn't pre-place the div)
  • STEP 1 Auction Initialized via CDN. Target: limelightDigital | Timeout: 1200ms
  • STEP 1.5 Requesting video ad payload from Limelight routing networks. Player size: XXXxYYY
  • Either 🏆 WINNER <bidder> | Raw CPM: $X.XX (real bid) or ⚠️ NO MARKET DEMAND (no bid)
  • STEP 3 Math Engine processing. Target CPM calibrated… (only on winning bid)
  • STEP 6 Dispatching VAST Request via Cloud Engine layer.

Network checklist

Open DevTools → Network. Verify each of these requests fires:

RequestExpected statusWhat to confirm
cdn.jsdelivr.net/.../engine/player.js200Engine loaded
cdn.jsdelivr.net/.../prebid/prebid.js200Prebid bundle loaded
imasdk.googleapis.com/.../ima3.js200IMA SDK loaded
POST {host}/ortbhb (e.g. ads-jbi003.rtba.bidsxchange.com)200OpenRTB bid request fired. Inspect Payload tab — should contain site, imp, video, ortb2 etc.
prebid.adnxs.com/pbc/v1/cache200Caches the winning VAST (only on a real winning bid)
GET pubads.g.doubleclick.net/gampad/ads?...200GAM ad call. Decode the cust_params query parameter — must contain hb_pb, hb_bidder, and (on a winning bid) hb_uuid.

Visual checklist

  • The video player renders at the expected size and position on the page.
  • A pre-roll ad plays before content (it will be Google's "Preroll 10 Seconds" test ad against the sample tag, or your bidder's real creative on a Limelight-allowlisted domain).
  • Content video plays after the ad completes.
  • Standard Video.js controls (play/pause, mute, volume, progress, fullscreen) appear and respond.
Decoding the GAM URL

To inspect cust_params easily, copy the full GAM gampad/ads request URL, paste it into a URL decoder (or run decodeURIComponent(decodeURIComponent(url)) in your console twice), and look for the cust_params value. It should be a &-separated list of key=value pairs including hb_pb, hb_bidder, and on winning bids hb_uuid.

Troubleshooting

No POST /ortbhb request in the Network tab

Means Prebid never fired the bidder. Possible causes:

  • Adapter not in Prebid build. Inspect pbjs.installedModules in the console — limelightDigitalBidAdapter must be present. If not, your data-prebid-url is pointing at a build without it.
  • Consent module blocking the auction. Look for "Canceling auction as per consentManagement config" in console. See Consent management.
  • Adapter rejected the ad unit. Look for auctionDebug messages with type WARNING. Common: "video ad unit containing bidders that don't support video" — your Prebid build is missing the adapter or its video support.
POST /ortbhb returns 200 but bid response is empty

Bidder responded but didn't bid. Almost always means:

  • Domain not allowlisted by Limelight. Verify with Limelight that your origin (e.g. https://www.example.com) is on the approved list. New allowlist entries can take 1–3 business days.
  • Floor below Limelight's threshold. If you've set a floor, lower it.
  • Ad slot size not supported. The engine sends the actual rendered mount size. If the player is unusually small (e.g. < 300×200), bidders may not bid. Check STEP 1.5 log for the size.
Bid wins but no ad plays / wrong ad plays / GAM falls through to house

The auction worked but the GAM side failed to resolve the bidder's creative. Check:

  • The GAM URL cust_params must contain hb_uuid. Decode the URL and confirm. If missing, you're on an old engine version (< v1.2.0) — upgrade.
  • Your line item targeting must match hb_pb exactly. Prebid emits "2.50", your line item must target "2.50", not "2.5".
  • Your VAST wrapper creative must use %%PATTERN:hb_uuid%%. Confirm in GAM → Delivery → Creatives → your wrapper. The macro must be present and spelled correctly.
  • Your Prebid Cache must be reachable. If you use a private cache, confirm it accepts requests from GAM's origin and has the cached entry.
Request timeout after 1200ms in console

Bidder didn't respond in time. Options:

  • Increase data-timeout to 1500 or 2000 ms. Industry standard for video is 1500–3000 ms.
  • Check Limelight's regional endpoint — if your traffic is far from ads-jbi003, ask Limelight if they have a closer host.
  • Inspect timeToRespond in pbjs.getEvents() → auctionEnd.args.bidderRequests. If consistently just over 1200, your network path is the bottleneck.
Ad plays but content doesn't resume / player stays paused after ad

IMA's CONTENT_RESUME_REQUESTED event didn't fire or was caught by a publisher's CSS rule. Check:

  • The mount div has no parent rule forcing pointer-events: none or opacity: 0 on the ad container.
  • The browser didn't suspend the page during the ad (tab backgrounded). Re-test in a foreground tab.
Player doesn't autoplay

Browser autoplay policies require muted=true for unattended autoplay. Confirm:

  • data-autoplay="true" AND data-muted="true" are both set.
  • The page is served over HTTPS (some browsers refuse autoplay on http://).
  • The user hasn't disabled autoplay globally in their browser settings.

On iOS Safari, additional restrictions apply (no autoplay until first user gesture in some contexts) — there is no workaround at the engine level.

Console error: currentScript missing anchor validation

The engine couldn't locate its own <script> tag. The fix is to ensure the snippet has id="adtech-player-core" attribute. If you regenerated the tag and stripped the ID, regenerate.

Mixed content warnings in the console

Your page is served over HTTPS but one of these URLs is http://:

  • data-video — the content video URL
  • data-tag — the GAM ad tag URL
  • A creative returned by Limelight or GAM that has an http resource inside

The browser will block these. Force all URLs to HTTPS.

FAQ

Can I run multiple players on the same page?

Not in the current engine. The integration assumes a single video ad unit per page. Multi-player support is on the v2 roadmap. Workaround for now: load the engine in an iframe per player.

Can I add other Prebid bidders besides limelightDigital?

You need a custom Prebid build that includes the additional adapters. Either ask your account manager to provide a build that includes the bidders you want, or build your own from github.com/prebid/Prebid.js with the relevant --modules flags.

The engine currently only registers one bidder (limelightDigital) per auction. Adding more bidders requires an engine code change — talk to your account manager.

What happens if Prebid fails to load?

The engine catches the failure and calls GAM directly with hb_pb=0.00&hb_bidder=none. Your house ad / fallback line item fills the impression. The publisher never loses an ad opportunity.

How do I track auction performance?

The current build doesn't include a Prebid Analytics adapter. To track auction-level metrics (bid latency, bid rate, win rate per bidder), request a Prebid build with your preferred analytics adapter (Magnite, RTBHouse, custom).

Impression-level metrics (eCPM, revenue, fill rate) are available natively in GAM Reports — filter by the hb_bidder key-value.

Will ad blockers affect this?

Yes, like all Prebid integrations. Common ad blockers (uBlock Origin, AdBlock Plus) will block: the engine script itself, prebid.js, the bidder endpoint, the GAM call, the IMA SDK, and the resulting ad. The engine has no special evasion behavior. Industry consensus is to detect ad blockers via a separate sniffer and either ask the user to disable it or serve content without ads.

What's the latency impact on my page?

The engine script is async and doesn't block page rendering. The auction adds ~1200 ms (the configured timeout) before the first ad starts playing — content video is already loaded and ready by then. Overall: zero impact on Time to Interactive; a configurable delay only on the pre-roll ad start.

Can I customize the player controls or skin?

The player is video.js v8 with the default skin. To customize, override video.js CSS by placing rules after the engine's <script> tag, targeting .video-js selectors. The engine doesn't expose a JS API for control configuration — file an issue if you need one.

How often does the player refresh the auction?

Once per page load. There is no automatic refresh. If you want a mid-roll ad after content, that's not currently supported by the engine.

What's the difference between @v1.2.2 and @main?

@v1.2.2 is an immutable tag — jsDelivr serves the exact bytes that were tested at release time and the URL is cached aggressively (1-year CDN TTL).

@main points at the latest commit on the main branch, cached for ~12 hours by jsDelivr. Use @main for development; always pin to a version tag for production.

Is this GDPR / CCPA compliant out of the box?

The engine includes the technical hooks for compliance (TCF v2 detection, USP detection, optional GPP). Whether your overall integration is compliant depends on: (1) you placing a registered CMP on your page in regulated jurisdictions; (2) your privacy policy disclosures; (3) your data processing agreements with bidders. Consult counsel — the engine does not replace a compliance program.
